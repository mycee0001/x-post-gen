# 日本語文章ベストプラクティス (X 投稿系すべての共通ルール)

このガイドは `/x-post`・`/x-quote`・`/x-reply` の **すべての本文生成** に必ず適用する。
個別の生成プロンプト (post_generation.md / quote_generation.md / reply_generation.md) を読むときは、このファイルも必ず一緒に読み込み、両方の制約に従うこと。

---

## 1. 行と文の長さ

- 一行は **最大 50 文字**、理想は 35 文字前後
- 句点までが長くなりそうなら、読点の位置で改行を入れる
- 例外: カギカッコ内の引用・固有名詞、または視覚的・心理的効果を狙う場合

【悪い例】
> 昨日の打ち合わせで、来期のプロダクト戦略について議論したのだが、結局結論が出ずに次回に持ち越しとなった。

【良い例】
> 昨日の打ち合わせで来期のプロダクト戦略を議論しましたが、
> 結論が出ずに次回へ持ち越しとなりました。

---

## 2. 文体（口調）

| 用途 | 文体 | 補足 |
|---|---|---|
| x-post (発信型) | **常体（だ・である調）をベース** | 読者を意識した説明や特定のメッセージを伝えたい場合は、部分的に丁寧体を許容。**段落単位で統一** すること |
| x-quote (受動型) | **丁寧体（です・ます）** | 引用元の話者を踏み台に発信するため、読み手への目線を保つ |
| x-reply (受動型) | **丁寧体（です・ます）** | 対話なので、より柔らかく |

- 一人称は **「私」** で統一（俺・僕・自分は使わない）

---

## 3. 使用禁止用語

下記のような語彙はいかなる文脈でも使用しない。

### 略式侮蔑語

キモい、キショい、ウザい、くっさ、くさ、は？、それらの語尾「い」を抜いた感嘆形（「キモ」「ウザ」など）

### 特定グループへのレッテル

エアプ、キッズ、陰キャ、陽キャ、オタク（属性ラベルとして揶揄する用法）、チー牛

### 差別用語

ガイジ、めくら、つんぼ、池沼、アスペ、統失、クロンボ、その他障害・人種・出自への侮蔑語

### 性的侮辱を含む用語

まんさん、ちんさん、弱者男性、クソオス、フェミ（蔑称用法）など

> 特に **略式侮蔑語と「特定グループへのレッテル」は手癖で出やすい** ので強く意識して避けること。

## 4. 使う単語のレベル

### 中学生でもわかる単語に置き換える

- 専門用語・略語（CPA, KPI, ARPU, NEAT, COGS, TAM, ROAS など）は、平易な言葉に置き換える
- どうしても専門用語を使う場合は、その意味を一文で添える
- カタカナ語の多用も避ける。日本語に置き換え可能なら置き換える

【悪い例】
> CPAを下げるためにオンボーディングのCVRを最適化する。

【良い例】
> 顧客一人を獲得する費用を下げるため、初回利用までの離脱を減らす設計を見直す。

### ADHD/発達特性領域の用語の扱い

- 当事者向け文脈（x-reply / x-quote）では、ある程度の専門語（実行機能・ワーキングメモリ・ニューロダイバーシティなど）は読者層に届くため使用可
- ただし「認知資源」「メタ認知」など抽象度が高い語は、文中で何を指すかが伝わるよう一文を補う

## 5. 表現技法（優先順位順）

下記の番号は影響度の高い順。**6 → 7 → 4 → 3 → 2 → 5 → 1** の順で守る。

### 6. 主述を対応させる（最優先）

【悪い例】私の夢は、社長になりたいです。
【良い例】私の夢は、社長になることです。

### 7. 修飾語は被修飾語の直前に置く

【悪い例】私は連休中、毎朝夫が書いた新聞連載記事を読み返した。
【良い例】私は連休中、夫が書いた新聞連載記事を毎朝読み返した。

### 4. 指示語を削る・具体化する

【悪い例】それを使えば作業が早くなります。
【良い例】このツールを使えば作業が早くなります。

### 3. 接続詞を削る

【悪い例】しかし、この方法には欠点があります。なぜなら、〜
【良い例】この方法には欠点があります。〜

### 2. 不要な修飾語を削る

「とても」「非常に」「本当に」「めちゃくちゃ」などは多用しない。

### 5. 重複表現を避ける

「頭痛が痛い」「まず最初に」「危険が危ない」「収入を稼ぐ」など。

### 1. 短く言い切る

語尾を「〜と思います」「〜と感じます」「〜気がします」に頼り過ぎない。
言い切る箇所と推測の箇所を意図的に切り替える。

## 6. 引用パターンの禁止 (x-reply / x-quote 共通)

### NG パターン

リプライ・引用ツイート本文の **冒頭で必ず元ポストの一文をカギカッコで引用する** スタイルは、パターン化して不自然になる。例えば下記は禁止：

【NG】
> 『〇〇は△△だ』、当事者向けプロダクトを作る側でも本当にそう感じます。

### OK パターン

- **自分の主張が元ポストの特定の語句に明確に紐づく場合に限り**、その語句に触れる
- 触れる場合も、冒頭にカギカッコ引用を貼り付けるのではなく、自分の言葉の中に自然に織り込む
- 元ポストへの言及は **無くてもよい**。むしろ自分の観察・経験・補足から書き始める方が自然な対話になる

【良い例 1：引用なしで自分の観察から始める】
> 環境を整えるだけで集中の質が変わる、という体感はADHD当事者にも共通します。
> 物理的に視界の刺激を減らせるかが、その日の処理能力の上限を決めますよね。

【良い例 2：引用元の特定語句に明確に紐づくときだけ、自然に織り込む】
> 「気合いより仕組み」という指摘、設計の観点でも揺るぎない原則だと思います。
> 意志は脳の予算を一気に食う高コスト機能なので、頼ると持続しません。

## 7. 自己ポジショニングの暗黙化（x-reply / x-quote 限定）

x-post（発信型）では「ADHD向けプロダクトを作っている立場として〜」のような **自己権威付け** は許容される。発信主体の専門性を読者に提示する型として有効だから。

一方 **x-reply / x-quote では、自己ポジショニングを地の文で名乗らない**。理由：

- 主役は **相手のポスト**（リプライ）または **広い読者の関心事**（引用）であって、発信者の自己紹介ではない
- 「〇〇する側として」「現場でも」と引き寄せる動きは、**他人の土俵から自分の土俵への強引な話題転換** で、対話の温度を冷ます
- 権威は表に出さず、自分の知識範囲を使って **役立つ考え・示唆** でポストに呼応する方が自然。専門性は **言葉の精度・観察の角度・機序の指摘** で滲ませる方が知性が読者に伝わる

### NG パターン（自己ポジショニング明示）

冒頭・中盤・文末いずれにも、下記のようなフレーズを **挿入しない**。

- 「ADHD向けアプリを作っている側でも」
- 「当事者向けプロダクトを作っている立場として」
- 「当事者向け設計でも軸になる考えです」
- 「現場でもよく見ます」「現場でも本当に通底する話ですね」
- 「私の専門領域でも」
- 「実装している側からの観察として」
- 「ADHD領域の研究でも近い知見が」← 自分が研究側の人間ではないので特に NG

### OK パターン（暗黙の専門性）

専門性は以下で示す：

1. **正確な専門用語の選択** — 「実行機能の予算を食う」「ワーキングメモリの再構築コスト」のように、その領域を理解している人だけが使う語彙の精度
2. **機序の指摘** — 「未完了が積もると自己効力感が削れる構造」「中断時に名詞で止まると再開時に状態を再構築するコストが乗る」のような原因→結果の解像度
3. **具体例の質** — 抽象論ではなく、その領域で観察される具体例（ただし「現場で見る」と地の文で語らず、現象そのものを書く）
4. **論点の接続** — 元ポストの主張を別の概念・枠組みに自然に接続する

### 書き換え例

【悪い例】
> 中断の仕方を変えるだけで再開速度が上がる、という指摘は、ADHD向け設計でも軸になる考えです。
> 名詞で止めると『何の状態だっけ』の再構築コストが毎回かかる。

【良い例】
> 中断の仕方を変えるだけで再開速度が上がる、という指摘は本質的だと思います。
> 名詞で止めると次回開いた時に『何の状態だっけ』の再構築コストが乗る。
> 動詞や次の一動作で止めれば、再開時の認知負荷が一段下がる構造ですね。

【悪い例】
> 実行機能の話、ADHD向けアプリを作っている側でも大きなテーマです。
> 特に返信は判断が累積する作業で、認知の予算を一気に食う。

【良い例】
> 返信は『次に何を書くか』『どの順で返すか』の判断が累積する作業で、
> 実行機能の予算を一気に食う構造ですよね。
> 入口の判断を一段減らすだけで行動量が変わる、というのは確かに観察できる現象です。

### 例外

x-reply / x-quote でも、相手から直接「あなたはどういう立場？」と聞かれた応答など、**自己言及が論理的に必要な場面では文末・1 回までに限定**して使ってよい。デフォルトは禁止。

## 8. 出力チェックリスト（自己点検）

本文を書き終えたら、最終出力前に以下を確認する：

- [ ] 冒頭がカギカッコ引用から始まっていないか（リプライ・引用 QT のみ）
- [ ] (x-reply / x-quote) **完全同意スタンスを保てているか** — 否定・反論・相対化・上から目線の指摘・敵視メタファーが混入していないか（第 9 節参照）
- [ ] (x-reply / x-quote) 自己ポジショニング明示句が混入していないか／主語と話題の中心が相手のポスト or 広い読者になっているか
- [ ] 一行 50 字超の行がないか
- [ ] 文体が用途に合っているか（post = 常体、quote/reply = 丁寧体）
- [ ] 一人称が「私」になっているか
- [ ] 禁止用語が混入していないか
- [ ] 専門用語・略語に説明が添えられているか、もしくは平易な語に置き換えられているか
- [ ] 主述が対応しているか
- [ ] 「とても」「本当に」「めちゃくちゃ」など修飾語の多用がないか
- [ ] 強い断定（「絶対」「必ず」「100%」「唯一」）がないか
- [ ] (x-reply / x-quote) **`agreement_check.py` で SAFE 判定を確認**したか

## 9. 同意スタンスの強制 (x-reply / x-quote 限定)

x-post（発信型）では論評・反論・批判的視点も許容される。

一方 **x-reply / x-quote では、対象ポストへの完全な同意・賛同のみが許される**。否定・反論・相対化・上から目線の指摘・敵視メタファーは、いかなる理由があっても禁止。

理由:
- リプライは元投稿者との関係構築。引用は元ポストを **足場** として広いオーディエンスに発信する形式。どちらも踏み台にされる元ポストへの否定的姿勢は、関係を壊す/読者の温度を冷ます/炎上の遠因になる
- 元ポストに違和感がある場合、応答するのではなく **その候補自体を不採用にして別候補に差し替える** のが正解。妥協して「ぼかして同意」する必要はない

### 許される姿勢 (4 種類のみ)

1. **同方向の補強** — 元ポストに同意し、機序・原理で裏打ちする
2. **同方向の具体化** — 元ポストに同意し、具体例・数字・事例で肉付けする
3. **同方向の展開 / パターン適用** — 同じ原理を別領域に拡張する(否定方向には絶対に振らない)
4. **建設的な深掘り質問** — 元ポストの主張を前提として、さらに掘り下げる(質問の前提条件で元ポストを否定しない)

### 否定スタンスの NG パターン

下記は本文のいかなる位置にも置かない。`agreement_check.py` で BLOCK 判定される:

- **比較相対化**: 「Xより Y のほうが」「X ではなく Y」
- **否定的評価語**: 「強引」「乱暴」「的外れ」「早計」「浅い」「表面的」「単純化しすぎ」「誇張」「飛躍」「違和感」「は強め」「はやや強い」「は荒々しい」など
- **訂正・打ち消し**: 「とは言えません」「とは限らない」「わけではない」「は誤解」「は誤り」「とは違う話」「とは別の話」
- **メタ批判**: 「という命名は強め」「という表現はやや強い」「という言い方はきつい」など、元ポストの言葉選び自体を批判
- **敵視メタファー**: 「を敵にする」「を悪者に」「を切り捨てる」「を一蹴する」
- **明示的反対**: 「私は反対」「賛成しかねる」「同意しかねる」「納得できない」「疑問が残る」「懐疑的」

### 同意の温度を下げる WARN パターン

下記も避ける(`agreement_check.py` で WARN 判定):

- **逆接からの書き出し**: 「しかし」「ただし」「とはいえ」「けれど」「むしろ」「逆に」「他方で」「一方で」を文頭に置く
- **限定・矮小化の前置き**: 「に偏ると」「に留まると」「だけでは」「だけ見ると」「を一面的に」「に寄り過ぎ」
- **上から目線の補足**: 「正確には」「厳密には」「補足すると」「言い直すと」「注意したいのは」「誤解されがち」
- **評価語+逆接**: 「強めですが」「乱暴ですが」「極端ですが」のように評価語+「が」で温度を下げる構造

### 書き換え例

【NG】
> スマホ認知症という命名は強めですが、本質は注意配分の癖の話。スマホを敵にするより連続した思考時間を測るほうが有効に思います。

【OK】
> スマホ認知症が指す現象の核は、情報の切り替え頻度が前頭前野の作業記憶を圧迫することにある、という整理がしっくりきます。連続した思考時間を1日で何分確保できているか、を測ることで状態が可視化できそうですね。

【NG】
> 当事者の語りが「家庭で否定された」前提に偏ると、特性の影響を切り分けにくくなりますね。

【OK】
> 理解的な親に育てられた当事者の話は、診断時期・自己受容のプロセス・進路選択など、まだ十分に語られていない別軸の知見が出てきそうですね。問いかけとして本当に意義深いと感じます。

### チェッカー

`agreement_check.py` を最終出力前に必ず通す:

```bash
python3 .claude/skills/_x-shared/scripts/agreement_check.py --text "<本文>"
```

- **BLOCK** → 再生成必須(最大 2 回)、ダメなら候補を差し替え
- **WARN** → 再生成して SAFE を目指す。再生成しても SAFE にならず、文意としては同意の方向であると確信できる場合にのみ警告付きで提示
